Improving Databricks Genie: Raising Natural-Language-to-SQL Accuracy

Genie's answers are only as good as the semantic context you give it. Most "Genie is wrong" tickets are not model problems — they are metadata and modeling problems. This is the end-to-end playbook for fixing them, with an architecture, the levers in priority order, and an evaluation harness to prove it worked.

Feodor Fernando• Lakehouse Analytics Research• ~13 min read•Rev. 2026-06

Abstract

Databricks Genie (AI-BI) translates business questions into SQL over Unity Catalog assets. Out of the box, accuracy on a real enterprise catalog is mediocre — not because the underlying model is weak, but because the catalog gives it almost nothing to ground on: cryptic column names, undefined business metrics, ambiguous join paths, and hundreds of overlapping tables. This paper treats NL-to-SQL accuracy as an engineering problem with measurable levers. We present a grounding architecture, rank the levers by empirical impact (semantic comments → certified metric views → curated surface area → example pairs), define an execution-match evaluation harness to quantify progress, and describe the feedback flywheel that compounds accuracy over time.

1. How Genie produces an answer

Conceptually Genie performs retrieval-augmented SQL generation: it assembles context about the available tables (names, column comments, sample values, certified metrics, curated instructions, and example question→SQL pairs), retrieves the slice relevant to the user's question, and prompts an LLM to emit SQL, which is then executed against the warehouse. Every point in that chain where context is thin or ambiguous is a place the model must guess — and guessing is where wrong answers come from.

3. Grounding architecture

The fix is to engineer the context Genie consumes. The diagram shows the grounding stack: a curated gold schema with rich semantics feeds Genie, and a feedback loop continuously improves that semantic layer from real usage.

4. Lever 1 — Describe the schema in business language

Highest ROI, lowest effort. Table and column COMMENTs are part of the context Genie reads; rich comments convert guessing into grounding.

COMMENT ON TABLE gold.revenue_daily IS
  'Daily revenue and engagement by region. One row per (date, region).
   Source of truth for revenue reporting. Excludes test accounts.';

ALTER TABLE gold.revenue_daily ALTER COLUMN revenue
  COMMENT 'Total gross revenue in USD = sum of order amounts for the date/region.';
ALTER TABLE gold.revenue_daily ALTER COLUMN active_users
  COMMENT 'Distinct users who transacted that day in that region.';

Two rules pay off most: rename cryptic columns where you can (comment them where you can't), and state the grain at table level ("one row per …"). Grain confusion is the top cause of silent double-counting.

5. Lever 2 — Certified metric views

Encode the correct business definition once, in a curated view, and point Genie at it. The model should never re-derive "7-day rolling revenue" — it should select from a view that already defines it.

CREATE OR REPLACE VIEW gold.metrics_revenue
  COMMENT 'Certified revenue metrics for AI-BI. Prefer this view for revenue questions.'
AS SELECT date, region, revenue, txn_count, avg_order_value,
     sum(revenue) OVER (PARTITION BY region ORDER BY date
       ROWS BETWEEN 6 PRECEDING AND CURRENT ROW) AS revenue_7d_rolling
   FROM gold.revenue_daily;

This collapses the space of "plausible but subtly wrong" SQL the model might otherwise generate, and gives the business a single auditable definition of each metric.

6. Lever 3 — Curate the surface area

Do not expose 400 tables to a Genie space. Grant the space access only to a small, certified gold schema. Fewer, well-described tables beat a sprawling catalog: less ambiguity, fewer wrong joins, faster planning, and a far smaller chance of the model selecting a raw/bronze table that looks superficially relevant.

7. Lever 4 — Instructions & example pairs

Genie spaces accept general instructions and example question→SQL pairs; treat them as a living asset:

  • State defaults: "Revenue means metrics_revenue.revenue. Default currency USD. Exclude test accounts (is_test = false). 'Last week' = previous ISO week."
  • Provide canonical example pairs for the top ~20 questions — they act as few-shot anchors that strongly bias the model toward correct patterns.
  • Specify preferred join paths and tables to avoid (raw/bronze, deprecated marts).

8. Measuring it: an execution-match eval harness

You cannot improve what you do not measure. Build a benchmark of question → expected-result pairs and score Genie on execution match — does the generated SQL return the same result set as a hand-written gold query? — rather than fragile string match.

def evaluate(genie, benchmark):
    passed = 0
    by_cause = Counter()
    for q in benchmark:
        sql      = genie.ask(q.question)
        got      = run(sql)
        expected = run(q.gold_sql)
        if frames_equal(got, expected):       # set-equality on results
            passed += 1
        else:
            by_cause[classify_failure(sql, q)] += 1   # grain/join/metric/filter
    return passed / len(benchmark), by_cause

Track accuracy over time, re-run after every metadata change, and use the by_cause histogram to decide which lever to pull next. A drop in "metric" failures after adding certified views is the proof your work landed.

9. The feedback flywheel

Capture every thumbs-down answer together with the analyst's corrected SQL. Triage weekly: each correction becomes one of three durable artifacts — a new comment, a new metric view, or a new example pair — and ideally a new row in the eval set so the fix is regression-protected. Accuracy is a flywheel, not a one-time setup; the catalog gets smarter with use.

10. Governance & safety

  • Unity Catalog permissions bound what any user's Genie session can read — row/column-level security still applies, so Genie cannot leak data the user couldn't query directly.
  • Certified vs experimental: mark which views are certified so the organisation knows which answers are trustworthy.
  • Human-in-the-loop for decisions: Genie accelerates analysis; it does not replace review for board-level or regulatory numbers.

11. Impact pattern & lever ranking

Typical impact ordering

In practice: (1) column/table comments → (2) certified metric views → (3) curating the exposed schema → (4) example pairs. The first two alone resolve the majority of "Genie got it wrong" tickets, because they directly attack the two biggest gaps (ambiguous schema and undefined metrics). The remaining levers harvest the long tail.

Business impact is twofold: analysts self-serve questions that previously queued behind the data team (cycle-time reduction), and the organisation gains a single certified definition of each metric (consistency and trust). The companion Databricks Lakehouse + Genie project implements this semantic layer end to end.

12. Implementation checklist

  1. Comment every gold table (state grain) and every column in business language.
  2. Publish certified metric views for the core KPIs; mark them certified.
  3. Restrict the Genie space to a small curated gold schema only.
  4. Author space instructions (defaults, join paths) + top-20 example pairs.
  5. Stand up an execution-match eval set; run it in CI / on every change.
  6. Capture 👎 + corrected SQL; triage weekly into comments/views/pairs + eval rows.
  7. Enforce Unity Catalog permissions; separate certified from experimental.
