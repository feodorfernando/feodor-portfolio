Improving Databricks Genie: Raising Natural-Language-to-SQL Accuracy

Genie's answers are only as good as the semantic context you give it. Most "Genie is wrong" problems are not model problems — they are metadata problems. Here are the levers that move accuracy, in order of impact.

Feodor Fernando• Lakehouse Analytics Research• Rev. 2026-06

1. Why Genie gets answers wrong

Databricks Genie (AI-BI) translates a business question into SQL over Unity Catalog tables. When it's wrong, the failure almost always traces to one of four gaps:

  • Ambiguous schema — column names like amt, flag, dt give the model nothing to ground on.
  • Undefined business metrics — "revenue", "active user", "churn" each have a precise definition the model can only guess at.
  • Join ambiguity — multiple plausible join paths between tables.
  • No feedback loop — wrong answers are never captured as training examples or corrections.

2. Lever 1 — Describe the schema in business language

The single highest-ROI change. Table and column COMMENTs are part of the context Genie reads. Rich comments turn guessing into grounding:

COMMENT ON TABLE gold.revenue_daily IS
  'Daily revenue and engagement by region. One row per (date, region).';

ALTER TABLE gold.revenue_daily ALTER COLUMN revenue
  COMMENT 'Total gross revenue in USD. Sum of order amounts for the date/region.';

Rename cryptic columns where you can; comment them where you can't. Add table-level notes describing grain ("one row per …") — grain confusion is a top cause of double-counting.

3. Lever 2 — Certified metric views

Encode the correct business definitions once, in a curated view, and point Genie at it. The model should never re-derive "7-day rolling revenue" — it should select from a view that already defines it:

CREATE OR REPLACE VIEW gold.metrics_revenue
  COMMENT 'Certified revenue metrics for AI-BI. Prefer this view for revenue questions.'
AS SELECT date, region, revenue, avg_order_value,
     sum(revenue) OVER (PARTITION BY region ORDER BY date
       ROWS BETWEEN 6 PRECEDING AND CURRENT ROW) AS revenue_7d_rolling
   FROM gold.revenue_daily;

This collapses the space of "plausible but subtly wrong" SQL the model might generate.

4. Lever 3 — Curated Genie "instructions" & example pairs

Genie spaces support general instructions and example question→SQL pairs. Treat these as a living asset:

  • State defaults: "Revenue means metrics_revenue.revenue. Default currency USD. Exclude test accounts (is_test = false)."
  • Provide canonical example pairs for the 20 most common questions — these act as few-shot anchors.
  • Specify preferred join paths and which tables to avoid (raw/bronze).

5. Lever 4 — Constrain the surface area

Don't expose 400 tables to a Genie space. Curate a small, certified gold schema and grant the space access only to it. Fewer, well-described tables beat a sprawling catalog: less ambiguity, fewer wrong joins, faster planning.

6. Measuring it: an NL-to-SQL eval harness

You can't improve what you don't measure. Build a benchmark of question → expected-result pairs and score Genie on execution match (does the generated SQL return the same result set as the gold query?) rather than string match:

for q in benchmark:
    sql      = genie.ask(q.question)
    got      = run(sql)
    expected = run(q.gold_sql)
    q.pass   = frames_equal(got, expected)   # set-equality on results
accuracy = mean(q.pass for q in benchmark)

Track accuracy over time and re-run after every metadata change. Bucket failures by cause (grain, join, metric, filter) to know which lever to pull next.

7. The feedback loop

Capture thumbs-down answers and the analyst's corrected SQL. Triage weekly: each correction becomes either a new comment, a metric view, or an example pair. Accuracy is a flywheel, not a one-time setup.

8. Results pattern & takeaways

Typical impact ordering

In practice the lever ranking is: (1) column/table comments → (2) certified metric views → (3) curating the exposed schema → (4) example pairs. The first two alone resolve the majority of "Genie got it wrong" tickets.

  • Genie accuracy is a metadata & semantics problem far more than a model problem.
  • Comments + certified views + a small curated schema do most of the work.
  • Measure with execution-match evals; close the loop on failures.
