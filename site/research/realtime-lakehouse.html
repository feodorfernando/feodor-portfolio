Real-Time Lakehouse Architectures: Streaming, CDC & the Table-Format Wars

Open table formats turned the data lake into a transactional, real-time store. This paper covers what they enable, how Iceberg / Delta / Hudi differ, the merge-on-read trade-off at the heart of streaming, and the operational realities — small files, delete accumulation, exactly-once — that decide whether a streaming lakehouse survives production.

Feodor Fernando• Lakehouse Research• ~14 min read•Rev. 2026-06

Abstract

The lakehouse pattern resolves the decade-long split between cheap-but-dumb data lakes and powerful-but-closed warehouses by adding an open table format — ACID commits, snapshot isolation, schema evolution, and row-level mutations — on top of object storage. This paper explains the metadata mechanics that make this possible, contrasts the three dominant formats (Apache Iceberg, Delta Lake, Apache Hudi), analyses the copy-on-write vs merge-on-read decision that governs streaming latency and cost, and details the maintenance regime (compaction, delete rewriting, snapshot expiry, orphan cleanup) without which a real-time lakehouse quietly degrades. A reference architecture and a 2026 format-selection guide close the paper.

1. Why the lakehouse exists

The data lake gave us cheap, open, infinitely scalable storage — but no transactions, no consistent reads, and no safe updates. The warehouse gave us ACID and performance — but locked data inside a proprietary engine and a separate copy. The lakehouse merges them: an open table format adds the warehouse's guarantees directly on top of the lake's storage, so a single copy of data serves BI, ML, and streaming, readable by many engines at once.

2. The metadata tree (what a commit actually is)

All three formats share the same core idea: a table is a tree of metadata over immutable data files, and a "commit" is a single atomic swap of one pointer. Understanding this explains every capability that follows.

A commit writes new data/delete files → new manifests → new metadata → then atomically moves the catalog pointer. Concurrent writers use optimistic concurrency: the loser detects the pointer moved and retries. Per-file column statistics in the manifests drive file pruning — the lakehouse equivalent of an index seek.

3. What makes a lakehouse "real-time"

  • Streaming ingestion — Spark/Flink micro-batches commit seconds-to-minutes after data is produced, each batch an atomic snapshot.
  • Change Data Capture (CDC) — database changes (via Debezium) replayed as row-level upserts/deletes, keeping a queryable mirror of operational state.
  • Row-level mutations — merge-on-read deletes make continuous updates/deletes cheap enough to apply per micro-batch.
  • Snapshot isolation — readers see a consistent table while writers stream into it; no half-written partitions.

4. Copy-on-write vs merge-on-read

The decision that defines your latency/cost profile, common to all three formats. Under copy-on-write (COW) an update rewrites whole data files — reads stay fast, writes are expensive. Under merge-on-read (MOR) an update writes a small delete file plus new rows — writes are cheap, but reads must merge data + delete files until compaction folds them in.

              write cost    read cost               best for
  COW         high          low                     append-mostly, batch, BI gold
  MOR         low           grows with deletes      streaming upserts / CDC silver
Rule for streaming: ingest with MOR for cheap per-batch commits, then compact aggressively to keep read amplification bounded. Use COW for stable, read-heavy gold tables.

Iceberg's differentiators: hidden partitioning (queries never reference partition columns, and the partition spec can evolve without rewriting history) and the broadest cross-engine reach via the REST catalog. Delta is strongest inside the Databricks ecosystem and has matured fast with deletion vectors and liquid clustering. Hudi was purpose-built for high-frequency upserts with strong record-level indexing.

6. Reference architecture

Bronze appends raw cheaply; silver applies MOR MERGE upserts to dedup and correct late data; gold holds compacted COW aggregates for fast BI. Maintenance runs out-of-band so it never blocks ingestion. This is the shape of the companion Iceberg Streaming Lakehouse and CDC → Iceberg projects.

7. The operational realities (where projects fail)

  • Small files. Streaming commits create millions of tiny files; query planning slows and storage LIST/GET costs explode. Compaction (rewrite_data_files to 128–256 MB targets) is non-negotiable — the single most important maintenance job.
  • Delete-file accumulation. Under MOR, every update adds a delete file; monitor the delete-file ratio and run rewrite_position_delete_files to bound read amplification.
  • Snapshot & manifest growth. expire_snapshots and rewrite_manifests keep metadata and planning fast while preserving a time-travel window.
  • Orphan files. Failed commits leave unreferenced files; remove_orphan_files reclaims the storage.
  • Exactly-once. Achieved via checkpoints + idempotent MERGE keyed on a business key (and LSN for CDC), not by trying to force it across the whole chain.

8. CDC into the lakehouse

CDC is the bridge between operational databases and the lakehouse. Debezium reads the source WAL/binlog and emits change events; a streaming job applies them as row-level upserts/deletes via MERGE, guarded by the source LSN so replay and out-of-order delivery are idempotent. Deletes are exactly why Hive-style tables fail at CDC and why MOR row-level deletes matter — they make WHEN MATCHED THEN DELETE a cheap delete file instead of a full-file rewrite. The initial Debezium snapshot and the streaming tail are handled identically (upsert), so the mirror is correct from the first event.

9. Choosing a format in 2026

  • Vendor-neutral, multi-engine, long horizon → Iceberg.
  • Databricks-centric shop → Delta (deepest native integration, liquid clustering, deletion vectors).
  • Extremely high-frequency upserts with record indexing → evaluate Hudi.
  • Convergence is real (cross-format reads / UniForm). Bet on the open catalog layer rather than locking to a single engine.

10. Business impact

A real-time lakehouse collapses the gap between an event happening and a decision being made: fraud flags, inventory, and operational dashboards move from "yesterday's batch" to minutes-old. It also removes duplicate storage and ETL copies (one governed copy serves BI, ML, and streaming), and the open format avoids warehouse lock-in. The cost is a real maintenance budget — teams that skip compaction trade a fast launch for a slow, expensive table six months later.
