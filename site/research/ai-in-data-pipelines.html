Pipelines × AI

Integrating Generative AI into Production Data Pipelines

Where LLMs genuinely earn their place in a data pipeline — and the architectural line you must not cross: never let a probabilistic model sit silently in the deterministic data path.

Feodor Fernando• Data Engineering Research• Rev. 2026-06

1. The framing problem

"Add AI to the pipeline" is usually proposed as if AI were another transform step. It is not. A data pipeline's core promise is determinism and reproducibility: the same input must produce the same output, every run. A generative model is, by construction, probabilistic. The design question is therefore not "can the LLM do this step?" but "is this a judgement task or a correctness task?"

Use LLMs for judgement and generation at the edges; keep enforcement and movement of data deterministic in the core.

2. A taxonomy of where AI helps

Use caseAI roleIn hot path?
Schema mapping / column matchingSuggest mappings between source & target schemasNo — human-reviewed
Data quality rule generationPropose typed expectations from a profileNo — rules then run deterministically
Semantic anomaly detectionFlag values that are "wrong" in meaning, not just rangeSide path / alerting
Pipeline documentation & lineage proseGenerate human-readable docs from metadataNo — offline
Natural-language → SQL / transformsDraft queries for analystsNo — generated code is reviewed/tested
Unstructured → structured extractionTurn text/PDFs into typed rowsYes, but with validation gates

3. The "propose vs enforce" pattern

The safest and most reusable pattern across all of the above is to split the LLM's proposal from the system's enforcement:

profile / context ──► LLM ──► structured proposal (tool-use JSON)
                                     │
                                     ▼
                       deterministic executor (SQL / pandas / Spark)
                                     │
                                     ▼
                              auditable PASS / FAIL

The LLM never directly mutates data or returns a verdict; it returns a typed object (via tool-use / structured output) that plain code then executes. A hallucinated rule becomes a bad rule you can see in a diff — not silently corrupted data. (This is exactly how the companion LLM Data Quality Agent project is built.)

4. Unstructured extraction — the one true hot-path use

The case where an LLM legitimately sits inside the data path is converting unstructured input (support emails, contracts, scanned forms) into structured rows. Even here, you wrap it in gates:

  • Schema-constrained output — force the model to emit JSON matching a strict schema (tool-use), so downstream typing never breaks.
  • Validation layer — reject/route rows failing referential or range checks to a dead-letter queue for human review.
  • Confidence + provenance — store the model, prompt version, and a confidence signal alongside each extracted row for auditability.
  • Idempotency — cache by content hash so re-runs don't re-spend tokens or produce drift.

5. Cost, latency & failure modes

  • Cost scales per row. An LLM call per record is fine at thousands/day, ruinous at millions/hour. Batch, cache aggressively, and reserve the model for rows that cheaper heuristics can't classify.
  • Latency is non-deterministic. Never put a synchronous LLM call on a critical streaming path with a tight SLA; use async side-paths.
  • Model drift. Pin model versions; a silent provider upgrade can shift outputs. Re-run an eval set before adopting a new model id.
  • Prompt injection via data. Untrusted text fed to an LLM can carry instructions. Treat extracted instructions as data, never as commands.

6. Evaluation: you can't ship what you can't measure

Any AI step needs a golden eval set and a metric before it goes near production: extraction accuracy / F1, rule precision-recall, or NL-to-SQL execution match. Wire the eval into CI so a prompt or model change that regresses quality fails the build — the same discipline you'd apply to any deterministic transform.

7. Reference: an AI-augmented pipeline

sources ─► ingest ─► [deterministic clean/dedup] ─► curated lakehouse ─► BI
                          │                               ▲
              (side) LLP: DQ rule proposal          (side) LLM: NL→SQL for analysts
                          │                               │
              (hot, gated) LLM extraction for unstructured sources only

8. Takeaways

  • AI belongs at the edges (proposal, generation, extraction, docs), not as a silent core transform.
  • Propose-then-enforce keeps determinism and auditability.
  • Gate every hot-path use with schema constraints, validation, and idempotency.
  • No AI step ships without an eval set in CI.
