Pipelines × AI

Integrating Generative AI into Production Data Pipelines

A reference architecture for putting LLMs to work in real data platforms — where they belong, where they must never sit, and the engineering scaffolding (gating, evaluation, cost control, observability) that turns a demo into a production system.

Feodor Fernando• Data Engineering Research• ~12 min read•Rev. 2026-06

Abstract

Generative AI is being bolted onto data pipelines faster than it is being engineered into them. The result is a wave of brittle systems where a probabilistic model silently sits on a path that the rest of the organisation assumes is deterministic. This paper presents a production reference architecture for generative AI in data pipelines, built on one organising principle — separate proposal from enforcement — and surrounds it with the operational concerns that decide whether such a system survives contact with real traffic: schema-constrained extraction, idempotency and caching, a cost model that scales, evaluation gates in CI, prompt-injection defences, and end-to-end observability of the AI step itself.

1. The framing problem

A data pipeline's core contract is determinism and reproducibility: the same input must yield the same output on every run, and any output must be explainable from its inputs and code. A generative model breaks both guarantees by construction — it is probabilistic, version-sensitive, and opaque. Teams that treat an LLM as "just another transform step" inherit non-determinism in the one place a data platform cannot tolerate it: the lineage between source and serving tables.

The correct question is therefore never "can the model do this step?" — modern models can do almost any single step in a demo — but "is this a judgement task or a correctness task, and what happens on the day the model is wrong?" Judgement tasks (mapping, classification of ambiguous inputs, summarisation, drafting) tolerate a human or a validator in the loop. Correctness tasks (moving money, deduplicating keys, enforcing constraints) do not.

Design rule: use LLMs for judgement and generation at the edges of the pipeline; keep the movement and enforcement of data deterministic in the core.

4. The core pattern: propose, then enforce

The single most reusable design across every judgement use case is to split the model's proposal from the system's enforcement. The LLM never mutates data and never returns a verdict; it returns a typed object (via tool-use / structured output) that plain, auditable code executes.

context / profile ──► LLM (tool-use) ──► structured proposal (validated JSON)
                                              │
                                              ▼
                          deterministic executor (SQL / pandas / Spark)
                                              │
                                              ▼
                               auditable PASS / FAIL + artifacts

This buys three properties that matter in production. Auditability: a wrong proposal becomes a bad rule visible in a diff, not silently corrupted data. Reproducibility: the executor is deterministic, so re-running the pipeline on the same proposal is stable. Reversibility: proposals are versioned artifacts you can roll back. The companion LLM Data Quality Agent project implements exactly this — Claude proposes typed rules via tool-use; pandas enforces them and exits non-zero on failure.

5. The one true hot-path use: gated extraction

Converting unstructured input (support emails, contracts, scanned forms, log lines) into structured rows is the case where an LLM legitimately sits inside the data path. Productionising it means wrapping the model in four gates:

  • Schema-constrained output. Force the model to emit JSON matching a strict schema (tool-use / structured outputs) so downstream typing never breaks. Reject anything that does not validate before it touches a table.
  • Validation layer. Apply referential, range, and format checks; route failures to a dead-letter table for human review rather than dropping or guessing.
  • Confidence & provenance. Persist the model id, prompt version, token usage, and a confidence signal alongside each extracted row, so every value is traceable to the exact decision that produced it.
  • Idempotency & caching. Key extractions by a content hash so re-runs are no-ops — this prevents both token re-spend and output drift on replay.
def extract(doc) -> Row | DeadLetter:
    key = sha256(doc.bytes)
    if key in cache: return cache[key]          # idempotent on content
    out = llm.extract(doc, schema=ROW_SCHEMA)   # tool-use, schema-constrained
    if not ROW_SCHEMA.validate(out):            # gate 1: structure
        return DeadLetter(doc, "schema")
    if not business_rules_ok(out):              # gate 2: semantics
        return DeadLetter(doc, "rules")
    row = Row(**out, _model=MODEL, _prompt=PROMPT_V, _conf=out.confidence)
    cache[key] = row
    return row

6. Orchestration: where the AI task lives

In an Airflow/Dagster DAG, AI work should be modelled as distinct, retryable tasks with their own SLAs, not hidden inside a transform. Practical guidance:

  • Isolate the AI task so its (variable) latency and failure modes do not block the deterministic critical path. Side-path tasks run async and write to their own tables.
  • Bounded concurrency & rate limits on the model client; a backoff + retry policy distinct from normal task retries (LLM errors are often transient/rate-limit).
  • Batch where possible. Aggregate records into batched prompts to amortise overhead and cost; never one synchronous call per row on a tight-SLA stream.
  • Checkpoint partial progress so a failed batch resumes without re-spending tokens on completed items.

7. The cost model (do the arithmetic before you ship)

LLM cost scales per record, which is the opposite of how cheap deterministic transforms scale. A back-of-envelope model decides feasibility:

monthly_cost ≈ rows/month × calls/row × (in_tokens·$in + out_tokens·$out)

At thousands of documents/day this is trivial; at millions of rows/hour it is ruinous. Levers, in order of impact: (1) don't call the model on rows cheaper heuristics can classify (route only the ambiguous tail to the LLM); (2) cache by content hash; (3) batch; (4) use a smaller/faster model tier (e.g. Haiku) for high-volume, simple classification and reserve a frontier model for the hard cases; (5) use prompt caching for the stable system/context prefix.

8. Reliability & failure modes

  • Model drift. Pin model versions; a silent provider upgrade can shift outputs. Re-run the eval set before adopting any new model id.
  • Non-deterministic latency. Never put a synchronous LLM call on a critical streaming path with a tight SLA; use async side-paths and timeouts with deterministic fallbacks.
  • Prompt injection via data. Untrusted text fed to an LLM can carry instructions ("ignore previous instructions and…"). Treat any instruction found in data as data, never as a command; isolate tool permissions; never let extracted text drive privileged actions.
  • PII leakage. Redact or tokenise sensitive fields before they reach an external model; record what was sent for audit.
  • Partial/garbage output. Schema validation + dead-lettering ensures a bad response degrades gracefully instead of poisoning a table.

9. Evaluation: you cannot ship what you cannot measure

Every AI step needs a golden eval set and a metric before it goes near production, and that eval must run in CI so a prompt or model change that regresses quality fails the build — the same discipline applied to any deterministic transform.

# CI gate: fail the build if extraction quality regresses
def test_extraction_quality():
    results = [extract(c.doc) for c in GOLDEN]
    f1 = field_level_f1(results, [c.expected for c in GOLDEN])
    assert f1 >= 0.92, f"extraction F1 regressed: {f1:.3f}"

Pick the metric that matches the task: field-level F1 for extraction, precision/recall for rule proposals, execution-match for NL→SQL. Bucket failures by cause so you know which part of the prompt or schema to fix next.

10. Observability of the AI step

Instrument the model call as a first-class pipeline component. Track, per run: token usage and cost, latency distribution, dead-letter rate (the leading indicator of quality decay), schema-validation failure rate, cache hit rate, and confidence distribution. Alert on dead-letter-rate spikes and cost anomalies. Log the model id and prompt version on every record so an incident can be traced to a specific prompt change.

11. Case study (representative)

Scenario. A B2B platform receives ~40k inbound supplier emails/month that analysts manually keyed into an orders table. We replaced manual entry with gated LLM extraction (schema-constrained output → validation → dead-letter), a heuristic pre-filter for obvious cases, and a content-hash cache.

Outcome. ~85% of emails auto-extracted straight-through; the ambiguous ~15% routed to a review queue rather than guessed; manual keying time reduced by roughly an order of magnitude; every extracted row carries provenance for audit. Crucially, because extraction is gated and dead-lettered, zero unvalidated rows entered the orders table — the failure mode is "needs review", not "wrong data in production".

(Numbers are representative of the pattern, not a specific client deliverable.)

12. Implementation checklist

  1. Classify each candidate use: judgement vs correctness; side-path vs hot-path.
  2. For judgement → implement propose-then-enforce with tool-use + a deterministic executor.
  3. For hot-path extraction → add the four gates (schema, validation, provenance, idempotency).
  4. Model it as an isolated, retryable orchestration task with rate limits + backoff.
  5. Build the cost model; add pre-filter, cache, batching, and tier routing.
  6. Pin model versions; redact PII; defend against prompt injection.
  7. Ship a golden eval set wired into CI.
  8. Instrument tokens, cost, latency, dead-letter rate, cache hits; alert on regressions.
