Data Quality · Observability

AI-Augmented Data Quality & Observability

LLMs are excellent at proposing what "good data" looks like and at triaging noisy alerts — but enforcement must stay deterministic. A design for trustworthy, AI-assisted data quality.

Feodor Fernando• Reliability Research• Rev. 2026-06

1. The two halves of data quality

Data quality splits into defining expectations and enforcing them. Defining is judgement-heavy ("an age of 199 is implausible"; "this column should look like an email") — a great fit for an LLM. Enforcing must be exact and reproducible — a job for plain code. Conflating the two is the central mistake.

2. The six dimensions

DimensionQuestion it answers
CompletenessAre required fields present / non-null?
ValidityDo values conform to type, range, format, allowed set?
UniquenessAre keys unique; any duplicates?
ConsistencyDo related fields agree (cross-column, cross-table)?
TimelinessIs data fresh within its SLA?
AccuracyDoes it match reality / a source of truth?

3. Where the LLM adds value

  • Rule proposal from a profile. Feed column stats + sample rows; the model proposes typed expectations (not_null, range, regex, allowed-set) — bootstrapping coverage far faster than hand-writing rules.
  • Semantic checks. Catch errors a range check can't: a "country" value of XX, a product name in the email field, a free-text status that contradicts a numeric flag.
  • Alert triage. Cluster and summarize a flood of failures into a few root causes ("all failures are the new ingestion source's date format").
  • Incident narratives. Turn a failed check + lineage into a readable summary for the on-call engineer.

4. The non-negotiable: propose, then enforce

column profile ──► LLM (tool-use) ──► typed rules JSON
                                          │
                                          ▼
                      deterministic engine (pandas / Spark / SQL)
                                          │
                                          ▼
                                 PASS / FAIL + violation rows

The model only ever emits rules; it never declares data "good". A hallucinated rule is visible in code review; it cannot silently pass bad data. This is the architecture of the companion LLM Data Quality Agent project — Claude proposes via tool-use, pandas enforces.

5. From quality checks to observability

Point-in-time checks catch known problems; observability catches the unknown. The five pillars:

  • Freshness — when did this table last update vs expected?
  • Volume — did row counts move within normal bounds?
  • Schema — did columns/types change unexpectedly?
  • Distribution — did value distributions drift (null rate, mean, cardinality)?
  • Lineage — what upstream change explains a downstream anomaly?

LLMs help here by explaining anomalies across these signals, not by deciding thresholds — statistical baselines decide; the model narrates and correlates.

6. Operationalizing

  • Run quality checks as a CI/orchestration gate (exit non-zero on failure) so bad data fails the pipeline, not the dashboard.
  • Route violations to a dead-letter table + alert, not silent drops.
  • Version rules and prompts; treat them as code with review.
  • Keep an eval set for the LLM proposal step so model changes don't quietly regress rule quality.

7. Takeaways

  • LLMs define and triage; deterministic code enforces.
  • Cover the six dimensions; graduate from checks to five-pillar observability.
  • Gate pipelines on quality; never let probabilistic output be the final verdict.
