AI-Augmented Data Quality & Observability

LLMs are excellent at proposing what "good data" looks like and at triaging noisy alerts — but enforcement must stay deterministic. A production design for trustworthy, AI-assisted data quality, plus the five-pillar observability that catches what point checks cannot.

Feodor Fernando• Reliability Research• ~12 min read•Rev. 2026-06

Abstract

Data quality work fails in two opposite ways: too little (no checks, silent corruption) or too brittle (thousands of hand-written rules nobody maintains). Generative AI offers a way out of the maintenance trap — it can propose sensible rules from a data profile and triage alert floods into root causes — but it must never be the thing that decides data is "good". This paper presents an architecture that splits AI judgement from deterministic enforcement, maps it onto the six classical quality dimensions, extends point-in-time checks into five-pillar observability for the unknown failures, and specifies how to operate it as a CI/orchestration gate.

1. The two halves of data quality

Quality splits into defining expectations and enforcing them. Defining is judgement-heavy ("an age of 199 is implausible"; "this column should look like an email") — a strong fit for an LLM. Enforcing must be exact and reproducible — a job for plain code. Conflating the two is the central mistake; it puts a probabilistic verdict on the critical path where you need a deterministic one.

2. The six dimensions

DimensionQuestionExample check
Completenessrequired fields present?not_null(email)
Validityconforms to type/range/format/set?age between 0 and 120
Uniquenesskeys unique; duplicates?unique(customer_id)
Consistencyrelated fields agree?ship_date >= order_date
Timelinessfresh within SLA?max(updated_at) within 1h
Accuracymatches reality / source of truth?reconcile to control total

3. Architecture: propose, then enforce

The model proposes typed rules from a profile; deterministic code enforces them; violations go to a dead-letter table and alerts; failures feed a CI/orchestration gate. The LLM is on a side path that generates artifacts, never on the path that passes data.

4. Where the LLM adds value

  • Rule proposal from a profile. Feed column stats + sample rows; the model proposes typed expectations (not_null, range, regex, allowed-set, max-null-fraction), bootstrapping coverage far faster than hand-writing rules — then a human reviews the diff.
  • Semantic checks. Catch errors a range check can't: a "country" value of XX, a product name in the email field, a status string that contradicts a numeric flag.
  • Alert triage. Cluster and summarise a flood of failures into a few root causes ("all failures stem from the new source's date format").
  • Incident narratives. Turn a failed check + lineage into a readable on-call summary, accelerating mean-time-to-resolution.

5. The non-negotiable, in code

The model only ever emits rules (typed JSON via tool-use); plain code enforces them. A hallucinated rule is visible in review and cannot silently pass bad data. This is exactly the companion LLM Data Quality Agent.

# LLM returns typed rules; the executor is deterministic
rules = llm.propose_rules(profile)          # tool-use → validated JSON
for r in rules:                             # plain, auditable Python
    result = run_check(df, r)               # not_null / unique / min / max / regex / in_set
    if result.status == "FAIL":
        dead_letter(result.violation_rows)
        failures.append(result)
sys.exit(1 if failures else 0)              # CI gate

6. From checks to observability — the five pillars

Point-in-time checks catch known problems; observability catches the unknown. The five pillars monitor a table's vital signs continuously:

PillarWatches for
Freshnessdid the table update when expected?
Volumedid row counts move within normal bounds?
Schemadid columns/types change unexpectedly?
Distributiondid null rate / mean / cardinality drift?
Lineagewhat upstream change explains a downstream anomaly?

Here the LLM explains anomalies across these signals — correlating a distribution shift with an upstream schema change — but statistical baselines, not the model, decide the thresholds. The model narrates; the math decides.

7. Operationalising it

  • Gate the pipeline: checks run as a CI/orchestration step that exits non-zero on failure, so bad data fails the build — not the dashboard the CFO is looking at.
  • Dead-letter, don't drop: route violating rows to a quarantine table with the failing rule attached, plus an alert. Silent drops are how data quietly disappears.
  • Version rules & prompts as code, with review — proposals are durable artifacts, not ephemeral model output.
  • Keep an eval set for the proposal step so a model/prompt change can't quietly regress rule quality (precision/recall on a labelled set).
  • Place checks at boundaries: on ingest (reject bad sources early) and at the gold/serving boundary (protect consumers).

8. Case study (representative)

Scenario. A platform onboarded a new partner feed that intermittently sent malformed records. Hand-written rules existed for the old feeds but not the new one.

With the pattern. Profiling the new feed and asking the model to propose rules generated a sensible baseline in minutes (not_null on keys, a country allowed-set, an email regex, a plausible age range). The deterministic gate caught the malformed batches on ingest, dead-lettered them, and alerted — and LLM triage summarised "92% of failures = invalid country codes from partner X", pointing straight at the fix. Bad data never reached gold.

(Representative of the pattern, not a specific client deliverable.)

9. Implementation checklist

  1. Profile each dataset; have the LLM propose typed rules via tool-use.
  2. Review proposals; store them in a versioned rules registry.
  3. Enforce deterministically (pandas/Spark/SQL); exit non-zero on failure.
  4. Dead-letter violations + alert; never silently drop.
  5. Add five-pillar observability (freshness/volume/schema/distribution/lineage).
  6. Use the LLM to triage/narrate failures; baselines decide thresholds.
  7. Keep an eval set for the proposal step; place gates at ingest and serving.
