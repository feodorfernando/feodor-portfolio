Database Performance Improvement: Indexing, Partitioning & Query Optimization at Scale

A methodical field guide to the techniques that actually move latency — from "read the plan" through indexing, statistics, and partitioning, to scaling beyond a single node — with the trade-off each one buys.

Feodor Fernando• Database Engineering Research• ~13 min read•Rev. 2026-06

Abstract

Most database performance work fails because it starts from intuition instead of evidence. This paper lays out a disciplined optimisation loop — measure with the execution plan, fix the cheapest high-impact cause first, re-measure — and then surveys the techniques in the order you should reach for them: fresh statistics, the right indexes, SARGable query shapes, and only then partitioning. It closes with the architectural decisions (read replicas, caching, and separating OLTP from analytics) that matter once single-node tuning is exhausted, and maps the same principles onto the columnar lakehouse.

1. The optimisation loop

Performance tuning is a loop, not a guess: reproduce → read the plan → form one hypothesis → change one thing → re-measure. Changing several things at once means you never learn which one worked. Keep a before/after latency number for every change; "it feels faster" is not a result.

2. Read the plan, not your assumptions

The execution plan is the ground truth. EXPLAIN (ANALYZE, BUFFERS) in Postgres, or the actual execution plan in SQL Server / Synapse, tells you where the rows are, where the time goes, and what access method the optimiser chose.

EXPLAIN (ANALYZE, BUFFERS)
SELECT region, sum(amount)
FROM orders
WHERE order_date >= '2026-01-01'
GROUP BY region;

Read it for four tells: a Seq Scan on a large table (missing or unused index); estimated vs actual rows wildly diverging (stale statistics); spills to disk in sorts/hashes (under-sized work_mem or a bad join order); and the widest node in the tree (where your time actually goes — optimise that, not the query you happened to suspect).

3. Statistics first (the cheapest fix)

The optimiser's plan is only as good as its row estimates. A classic incident is a good plan going bad overnight after a bulk load left statistics stale, so the optimiser now picks a nested loop where a hash join is needed. Before touching indexes, refresh statistics (ANALYZE / update statistics) and tune autovacuum/auto-stats thresholds on high-churn tables. It is free and frequently sufficient.

4. Indexing — the biggest lever, with a cost

B-treeequality, range, orderingthe default workhorse
Compositemulti-column predicatesobeys left-prefix rule; order by selectivity/access

Partial / filteredhot subset onlye.g. WHERE status='active'; small & fast

The left-prefix rule is the most misunderstood: an index on (a, b) serves WHERE a=… and WHERE a=… AND b=…, but not WHERE b=… alone. Order composite columns by how queries actually filter.

Every index speeds reads but slows writes and costs storage. Index for your real read patterns; periodically audit and drop unused indexes.

5. Write SQL the optimiser can help

  • SARGable predicates. WHERE order_date >= '2026-01-01' uses an index; WHERE year(order_date) = 2026 wraps the column in a function and forces a scan. Keep the indexed column bare on one side of the comparison.
  • Avoid implicit casts. Comparing a varchar column to an integer silently disables index use — a frequent, invisible performance bug.
  • Select only needed columns. SELECT * defeats covering indexes and inflates I/O and network.
  • Set-based, not row-by-row. Replace N+1 patterns and correlated per-row subqueries with joins or window functions.
  • Filter early. Push predicates down before joins and aggregations so the engine carries fewer rows through the plan.

6. Partitioning — for tables too big to index your way out of

At hundreds of millions of rows, partitioning lets the engine prune whole partitions before scanning anything.

  • Range partition by date (most common): time-bounded queries touch only relevant partitions, and old partitions can be dropped/archived in O(1) instead of a giant DELETE.
  • Hash partition to spread write hotspots across partitions.
  • Don't over-partition. Thousands of tiny partitions add planning overhead; match granularity to query and retention patterns.

7. Beyond a single node

  • Connection pooling (PgBouncer / built-in pools): connection churn is a hidden tax that shows up as latency under load.
  • Read replicas to offload reporting/analytical reads from the OLTP primary.
  • Caching: materialised views, result cache, or an external cache (Redis) for expensive repeated aggregates — always with an explicit invalidation strategy.
  • Right engine for the job: an OLTP database for transactions, a columnar warehouse/lakehouse for analytics. Forcing heavy analytics onto an OLTP primary is the single most common self-inflicted performance wound.

8. The same principles in the lakehouse

Columnar lakehouse engines (Spark/Trino over Iceberg/Delta) re-express these ideas at the file layer. Partitioning + per-file column statistics enable file pruning (the analogue of partition pruning + index seeks); OPTIMIZE/compaction and Z-ordering / clustering co-locate related rows to minimise files scanned; and predicate pushdown is the SARGability of the lake. The mental model transfers directly — see the companion Real-Time Lakehouse paper.

9. A tuning checklist

  1. Reproduce with a real query + EXPLAIN ANALYZE.
  2. Refresh stale statistics first (cheap, often sufficient).
  3. Add/adjust the right index; confirm the plan actually uses it.
  4. Make predicates SARGable; remove implicit casts; trim selected columns.
  5. Partition only if table size demands it; size partitions to the workload.
  6. Pool connections; offload analytics to a replica or the right engine.
  7. Cache expensive repeated reads with a clear invalidation story.
  8. Re-measure and record before/after latency.

10. Business impact

Performance work is not academic: faster queries cut cloud compute bills (you pay for the scan), shrink dashboard load times (analyst productivity), and protect the OLTP primary from the reporting load that causes user-facing slowdowns. The highest-leverage single move in most shops is separating analytics from the transactional database — it simultaneously fixes latency on both sides.
