Database Performance Improvement: Indexing, Partitioning & Query Optimization at Scale

A field guide to the techniques that actually move latency — starting from "read the plan" and ending at partitioning, with the trade-offs each one buys.

Feodor Fernando• Database Engineering Research• Rev. 2026-06

1. Start with the plan, not the guess

The first rule of tuning: never optimize from intuition. Read the execution plan. EXPLAIN (ANALYZE, BUFFERS) (Postgres) or the actual execution plan (SQL Server / Synapse) tells you the truth — where the rows are, where the time goes, and whether the optimizer chose a scan, seek, or hash join.

EXPLAIN (ANALYZE, BUFFERS)
SELECT region, sum(amount)
FROM orders
WHERE order_date >= '2026-01-01'
GROUP BY region;

Look for: Seq Scan on a big table (missing index?), rows estimate vs actual wildly off (stale statistics), and spills to disk (work_mem too small / bad join order).

2. Indexing — the biggest single lever, with a cost

  • B-tree for equality and range predicates and for ordering. The default, the workhorse.
  • Composite indexes obey the left-prefix rule: an index on (a, b) serves WHERE a=… and WHERE a=… AND b=…, but not WHERE b=… alone. Order columns by selectivity / access pattern.
  • Covering indexes (INCLUDE columns) let a query be answered from the index alone — no heap/table lookup.
  • Partial / filtered indexes index only the hot subset (WHERE status = 'active'), keeping the index small.
Every index speeds reads and slows writes (and costs storage). Index for your read patterns; audit and drop unused indexes.

3. Keep statistics fresh

The optimizer's choices are only as good as its row estimates. Stale stats after a big load are a classic cause of a good plan suddenly going bad. ANALYZE / update statistics after bulk operations; tune autovacuum/auto-stats thresholds for high-churn tables.

4. Partitioning — for tables too big to index your way out of

When a table reaches hundreds of millions of rows, partitioning lets the engine prune whole partitions before scanning:

  • Range partition by date (most common) so time-bounded queries touch only relevant partitions, and old partitions can be dropped/archived cheaply.
  • Hash partition to spread write hotspots across partitions.
  • Don't over-partition: thousands of tiny partitions add planning overhead. Match partition granularity to query and retention patterns.

In the lakehouse the analogue is Iceberg/Delta partitioning + file pruning via column stats — same principle, different layer (see the Real-Time Lakehouse paper).

5. Writing queries the optimizer can help

  • SARGable predicates: WHERE order_date >= '2026-01-01' uses an index; WHERE year(order_date) = 2026 wraps the column in a function and forces a scan. Keep the indexed column bare.
  • Select only needed columns — SELECT * defeats covering indexes and bloats I/O.
  • Avoid N+1 / per-row subqueries — rewrite as set-based joins or window functions.
  • Filter before you join/aggregate; push predicates down.
  • Beware implicit casts (e.g. comparing a varchar column to an int) — they silently disable index use.

6. Beyond a single node

  • Connection pooling (PgBouncer / built-in pools) — connection churn is a hidden tax under load.
  • Read replicas to offload analytical/reporting reads from the OLTP primary.
  • Caching (materialized views, result cache, Redis) for expensive, repeated aggregates — with a clear invalidation story.
  • Right engine for the job: OLTP database for transactions, columnar warehouse/lakehouse for analytics. Forcing analytics onto an OLTP primary is the most common self-inflicted performance wound.

7. A tuning checklist

  1. Reproduce with a real query + EXPLAIN ANALYZE.
  2. Fix stale statistics first (cheap, often sufficient).
  3. Add/adjust the right index; verify the plan actually uses it.
  4. Make predicates SARGable; trim selected columns.
  5. Partition only if the table size demands it.
  6. Offload analytics to the right engine / replica.
  7. Re-measure. Keep a before/after latency number.

8. Takeaways

  • Measure with plans; never tune blind.
  • Indexes and fresh statistics resolve most problems before you reach for partitioning.
  • Write SARGable, set-based SQL — the optimizer can only help queries that let it.
  • Separate OLTP and analytics workloads onto the engines built for them.
